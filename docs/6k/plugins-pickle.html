Модули plugin-sdk, common-plugins, test-runner

1. plugin-sdk

SDK Библиотека для создания плагинов тестирования кода

Назначение

Библиотека, предоставляющая API для создания плагинов тестирования кода на различных языках программирования. Модуль содержит интерфейс LanguageTester и инструменты для парсинга отчётов о тестировании в формате JUnit XML.

Структура модуля

plugin-sdk/
├── src/main/java/org/supremecode/pluginsdk/
│ ├── LanguageTester.java # Главный интерфейс плагина
│ ├── JunitParser.java # Парсер JUnit XML отчётов
│ └── result/
│ │ ├── TestExecutionResult.java # DTO с результатами прогона
│ │ ├── TestCase.java # DTO с информацией о тест-кейсе
│ │ └── Summary.java # Пустая заглушка
│ └── junit/ # JAXB классы (сгенерированы из XSD)
│ ├── Testsuites.java
│ ├── Testsuite.java
│ ├── Testcase.java
│ ├── Failure.java
│ ├── Error.java
│ ├── Properties.java
│ ├── Property.java
│ └── ObjectFactory.java
└── pom.xml

Интерфейс LanguageTester

Центральный элемент SDK определяет контракт для всех плагинов тестирования:

public interface LanguageTester { String languageId(); // Идентификатор языка (java, cpp, javascript) String imageName(); // Название Docker-образа для запуска тестов String testsPath(); // Путь к файлу с тестами внутри контейнера String solutionPath(); // Путь к файлу с решением внутри контейнера String reportPath(); // Путь к файлу с JUnit отчётом TestExecutionResult verdict(String report, int exitCode); // Анализ результата }

Парсер JUnit XML (JunitParser)

Реализует unmarshalling XML-отчётов стандарта JUnit в объектную модель:

public class JunitParser { protected JAXBContext jaxbContext; public Testsuite parseTestsuite(String xmlReport) throws JAXBException, IOException public Testsuites parseTestsuites(String xmlReport) throws JAXBException, IOException }

DTO классы результатов

TestExecutionResult

Агрегированный результат прогона

record TestExecutionResult( int total, // Общее число тестов int failures, // Число проваленных тестов int errors, // Число ошибок выполнения boolean solved // Флаг успешного решения )

TestCase

Детальная информация о каждом тесте

record TestCase( String id, String name, String status, String message, String errorType, String errorMessage, String errorDetails, long executionTimeMs, String stdout, String stderr )

JAXB модель JUnit

Классы пакета org.supremecode.pluginsdk.junit являются результатом code generation из XSD-схемы jenkins-junit.xsd. Схема определяет иерархию элементов:

testsuites— корневой контейнер (опциональный)
testsuite— набор тестов (name, tests, failures, errors, time...)
testcase— отдельный тест (name, assertions, time, classname...)
failure— провал теста (type, message, текст ошибки)
error— ошибка выполнения (type, message)
properties— контейнер свойств набора тестов
property— пара name/value для метаданных
system-out/err— стандартный вывод

Зависимости pom.xml

2. common-plugins

Plugins Эталонные реализации LanguageTester для поддерживаемых языков

Назначение

Библиотека с эталонными реализациями интерфейса LanguageTester для поддерживаемых языков программирования. Содержит три плагина: Java, C++, JavaScript.

Структура модуля

common-plugins/
├── src/main/java/org/supremecode/commonplugins/
│ ├── JavaLanguageTester.java # Плагин для Java (JUnit 4)
│ ├── CppLanguageTester.java # Плагин для C++ (GoogleTest)
│ └── JsLanguageTester.java # Плагин для JavaScript (Jest)
└── pom.xml

JavaLanguageTester

Maven Surefire Plugin

imageName: sc-java-test
testsPath: /src/test/java/JunitTest.java
solutionPath: /src/main/java/org/example/Solution.java
reportPath: /usr/app/target/surefire-reports/TEST-JunitTest.xml

CppLanguageTester

GoogleTest + CMake

imageName: sc-cpp-test
testsPath: /sc_test.cc
solutionPath: /solution.hpp
reportPath: /usr/app/build/junit.xml

JsLanguageTester

Jest

imageName: sc-js-test
testsPath: /test.js
solutionPath: /solution.js
reportPath: /usr/app/junit.xml
Отличие JavaScript плагина: использует parseTestsuites() (обёртка для множества наборов тестов), тогда как Java и C++ используют parseTestsuite() для одиночного набора.

Механизм ServiceLoader

Класс LanguagePluginService в test-runner использует java.util.ServiceLoader для автоматического обнаружения реализаций LanguageTester:

public class LanguagePluginService { Map<String, LanguageTester> map = new HashMap<>(); LanguagePluginService() { ServiceLoader<LanguageTester> loader = ServiceLoader.load(LanguageTester.class); for (LanguageTester service : loader) { map.put(service.languageId(), service); } } }

Для автоматического обнаружения необходимо создать файл:

META-INF/services/org.supremecode.pluginsdk.LanguageTester

3. test-runner

Runner Микросервис для тестирования пользовательских решений

Назначение

Микросервис, выполняющий тестирование пользовательских решений в изолированных Docker-контейнерах. Является потребителем Kafka для получения задач и публикует результаты обратно в Kafka. Взаимодействует с MinIO для получения тестов и загрузки результатов.

Технологический стек

Spring Boot 3.2.3 Spring Kafka Docker Java 3.3.4 MinIO Java SDK 8.5.7 Kotlin 1.9.24

Структура модуля

Конфигурация Docker

DockerClient создаётся с кастомным HTTP-клиентом (Apache 5): .withDockerHost("tcp://localhost:2375") // Windows // или unix:///var/run/docker.sock для Linux .maxConnections(100) .connectionTimeout(Duration.ofSeconds(30)) .responseTimeout(Duration.ofSeconds(45))

Listener (Kafka Consumer)

Основной обработчик входящих сообщений:

@KafkaListener(topics = "test-topic", groupId = "test-group") protected void listen(@Payload TestMessage testMessage, ...) { // 1. Получает тесты из MinIO (bucket: problems) // 2. Получает решение из MinIO (bucket: solutions) // 3. Находит LanguageTester по languageId // 4. Создаёт Tester и запускает test() // 5. Сохраняет logs.txt и report.txt обратно в MinIO // 6. Публикует TestResultMessage в resultTopic }

Tester (Container Orchestration)

1. Создание архива (createArchive)

Формирует TAR-файл, содержащий файл тестов и файл решения. Использует TarArchiveOutputStream из Apache Commons Compress.

2. Создание контейнера (createContainer)

Memory: 400 MB
Swap: 1 GB
CPU: 1 ядро
Network: none (изоляция)

3. Процесс тестирования (test)

1. Создать контейнер
2. Скопировать TAR-архив в /usr/app
3. Запустить контейнер
4. Подписаться на логи
5. Дождаться или timeout (12 сек)
6. Скопировать отчёт
7. Вызвать verdict()
8. Убить и удалить контейнер

DTO TestResult

data class TestResult( val total: Int, // Всего тестов val failures: Int, // Провалы val errors: Int, // Ошибки val solved: Boolean, // Решение принято val statusCode: Int, // Код завершения (-1 если timeout) val report: String, // JUnit XML отчёт val logs: String, // Логи контейнера )

XSD Code Generation

В pom.xml настроен плагин Apache CXF XJC для генерации JAXB классов из XSD:

<plugin> <groupId>org.apache.cxf</groupId> <artifactId>cxf-xjc-plugin</artifactId> <version>4.0.0</version> <configuration> <sourceRoot>${basedir}/target/generated-sources/xsd</sourceRoot> <xsdDir>${basedir}/src/main/resources/xsd</xsdDir> <packagename>net.danil.generated.junit</packagename> </configuration> </plugin>
Генерирует классы в target/generated-sources/xsd/net/danil/generated/junit/, однако используются классы из org.supremecode.pluginsdk.junit.

Взаимодействие с MinIO

GET: GetObjectArgs.builder().bucket("problems").object(...)
PUT: PutObjectArgs.builder().bucket("solutions").object(...).stream(...)

Паттерны проектирования

Plugin Pattern
LanguageTester — точка расширения для новых языков
ServiceLoader Pattern
Динамическая загрузка плагинов без конфигурации
Strategy Pattern
Выбор LanguageTester по languageId
Template Method
Tester.test() определяет общую последовательность
Callback/Observer
LogCallback, WaitCallback для асинхронных событий Docker
DTO/Record
Неизменяемые объекты данных

Расширяемость

Для добавления нового языка необходимо:

1
Создать реализацию LanguageTester в отдельном модуле
2
Добавить JAR в classpath test-runner
3
Создать файл META-INF/services/org.supremecode.pluginsdk.LanguageTester с полным именем класса
4
Подготовить Docker-образ с тестовым фреймворком для языка
