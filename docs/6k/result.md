# Kafka

> Проблема, которую вы описали, действительно существует: при использовании Kafka максимальный параллелизм обработки
> сообщений в рамках одной группы потребителей ограничен количеством партиций топика. Если вы хотите динамически
> масштабировать число экземпляров test-runner (подов) в зависимости от нагрузки, то при превышении количества подов над
> партициями часть из них будет простаивать. Увеличение числа партиций возможно, но это необратимая операция (уменьшить
> нельзя без пересоздания топика), и она требует осторожности, особенно если важен порядок обработки сообщений. Кроме
> того, автоматическое масштабирование партиций вместе с подами — нетривиальная задача, которую обычно не решают
> встроенными средствами.

# Вопросы 

## Уровень защиты

Какой у нас уровень защиты от обхода тестирования решения и получения необоснованной оценки?
- строго защищённая система (изоляция процессов)
- или упрощённая (допускаем потенциальные обходы)

В первом случае не знаю как внедрить использование юнит-тестов. Кажется, оно доступно только во втором случае.

Сейчас у меня отсутствует изоляция между тестами и пользовательским кодом.
Файл с результатами тестов фактически не является доверенным источником, потому что формируется в недоверенном пространстве.
Возникает конфликт из-за того, что вывод тестового фреймворка я считаю доверенным, а запускается он вместе с
пользовательским кодом. 

Пример (Node.js)
Пользователь может изменить поведение стандартных проверок после загрузки тестов:

```js
const assert = require('assert');
assert.strictEqual = () => {
}; // тесты больше ничего не проверяют
```

Пример (C++)
Фоновый поток перезаписывает результат после тестов:

```cpp
std::thread([]{
std::this_thread::sleep_for(std::chrono::milliseconds(10));
std::ofstream("junit.xml") << "all passed";
}).detach();
```

## Модель хранения и передачи данных с kubernetes Job

Нужно согласовать решение, как
- загружать задачу и решение в Job
- забирать результат

Есть несколько конкурирующих подходов:

- ephemeral volume + initContainer и скачивание через api моего сервера + (как-то забрать результат, не знаю как. может `kubectl cp`)
- ephemeral volume + копирование через `kubectl cp`
- persistent volume независимый от жизненного цикла Job (lifecycle и очистка ресурсов полностью на нашей стороне, и наверное медленнее)

Больше адекватных решений не нашел. Хотя может отказался от каких-то путёвых.
