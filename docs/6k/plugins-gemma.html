Схема взаимодействия модулей Code Testing

Документация основана на файле docs/6k/plugins.md и описывает архитектуру системы для тестирования пользовательских решений в различных языках программирования.

📚 1. Plugin SDK (Библиотека API)

Предоставляет основной контракт (`LanguageTester`) и инструменты для парсинга результатов тестирования в формате JUnit XML.

🧩 Структура модуля

plugin-sdk/
├── src/main/java/org/supremecode/pluginsdk/
│   ├── LanguageTester.java          // Главный интерфейс плагина
│   ├── JunitParser.java             // Парсер JUnit XML отчётов
│   └── result/
│       ├── TestExecutionResult.java // DTO с результатами прогона
│       └── TestCase.java            // DTO с информацией о тест-кейсе
└── pom.xml

📌 Ключевые элементы

Интерфейс LanguageTester

Определяет контракт, который должен реализовать каждый плагин:

public interface LanguageTester {
    String languageId();           // Идентификатор языка (java, cpp, javascript)
    String imageName();            // Название Docker-образа для запуска тестов
    String testsPath();            // Путь к файлу с тестами внутри контейнера
    String solutionPath();         // Путь к файлу с решением внутри контейнера
    String reportPath();           // Путь к файлу с JUnit отчётом
    TestExecutionResult verdict(String report, int exitCode); // Анализ результата
}

DTO классы результатов

Неизменяемые объекты для структурирования данных:

  • TestExecutionResult: Агрегированный результат прогона (total, failures, errors, solved).
  • TestCase: Детальная информация о каждом тесте (name, status, message, executionTimeMs и др.).

🛠️ Парсер JUnit XML

Класс JunitParser отвечает за преобразование сырого XML-отчёта в объектную модель Java/Kotlin с использованием JAXB.

🚀 2. Common Plugins (Эталонные реализации)

Библиотека с готовыми реализациями `LanguageTester` для поддерживаемых языков: Java, C++, JavaScript.

Плагин Язык и фреймворк Docker-образ (imageName) Тип отчёта Основные пути в контейнере
JavaLanguageTester Java / JUnit 4 sc-java-test Surefire XML Tests: /src/test/java/..., Report: ...surefire-reports/TEST-...xml
CppLanguageTester C++ / GoogleTest sc-cpp-test JUnit XML Tests: /sc_test.cc, Report: /usr/app/build/junit.xml
JsLanguageTester JavaScript / Jest sc-js-test Jest/JUnit XML Tests: /test.js, Report: /usr/app/junit.xml

🔄 Механизм работы

Каждый плагин реализует общую логику `verdict()` и использует механизм ServiceLoader (для автоматического обнаружения) для интеграции в основной тестовый раннер.

⚙️ 3. Test Runner (Микросервис)

Центральный сервис, который управляет циклом тестирования: принимает задачу через Kafka, запускает Docker-контейнер и возвращает результат.

🏗️ Архитектура и Стек

  • Фреймворк: Spring Boot (Kotlin)
  • Взаимодействие: Apache Kafka (Consumer/Producer), MinIO (Хранение артефактов)
  • Инструменты: Docker API (Java), Kotlin, Jackson/Spring JSON

📋 Основные этапы работы

  1. Получение задачи: Listener подписывается на топик test-topic в Kafka.
  2. Подготовка артефактов: Получает тесты из MinIO (bucket: problems). Использует TarArchiveOutputStream для упаковки тестового и решения файлов в один архив.
  3. Управление контейнером (Tester):
    • Создаёт Docker-контейнер с заданными ресурсами (1 ядро, 400MB RAM).
    • Копирует архив в контейнер.
    • Запускает контейнер и ждёт его завершения или срабатывания таймаута (12 секунд).
    • Сбор логов и отчётов из MinIO.
  4. Анализ результата: Вызывает languageTester.verdict(report, exitCode) для преобразования сырого отчёта в структурированный объект TestResult.
  5. Публикация результата: Отправляет финальный результат (TestResultMessage) обратно в топик test-result-topic Kafka.

💡 Ключевые Паттерны Проектирования

  • Plugin Pattern: Определен через интерфейс LanguageTester.
  • ServiceLoader Pattern: Используется в LanguagePluginService для динамического обнаружения всех доступных плагинов без жёсткой привязки конфигурации.
  • Strategy Pattern: Выбор конкретной стратегии тестирования (Язык) по `languageId`.
  • Callback/Observer: Использование LogCallback и WaitCallback для асинхронного отслеживания состояния Docker-контейнера.
