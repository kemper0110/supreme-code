# Проектирование информационной системы

## Описание базового пользовательского пути

Начинаем с фронта.
Пользователь авторизуется.
Просматривает список задач.
Выбирает задачу и изучает описание.
Выбирает язык и программирует решение.
Отправляет решение на проверку и через время получает результат.
Если результат не удовлетворяет пользователя, он может изменить решение и отправить его на проверку еще раз.

## 2.2 Описание сущностей предметной области

- Пользователь: идентификатор, никнейм, email, логин, пароль в хэшированном виде, привилегии, роли, дата регистрации;
- Задача: идентификатор, название, описание, автор, сложность, теги, языки, шаблон решения, авто-тесты, эталонное
  решение;
- Тег: идентификатор, название;
- Решение: идентификатор, задача, автор, язык, код;
- Результат проверки: идентификатор, решение, код завершения, число тестов, число провалов, число ошибок, засчитано ли
  решение;
- Язык: идентификатор, название, путь до иконки, идентификатор языка в редакторе, расширение файла.

### Сервис проверки решения test-runner

Используем клиент-серверную архитектуру, как указано в ТЗ.
Обработка запроса на проверку решения.
Решение сохраняется в MinIO. Метаданные в БД.
Проверкой решения будет заниматься сервис test-runner.

Чтобы не потерять пользовательский запрос на проверку решения, добавим между web-server и test-runner очередь сообщений.
Очередью будет брокер сообщений Apache Kafka.
Это гарантирует, что запрос на проверку не потеряется, если сервисы проверки временно недоступны или перегружены.
web-server кладёт задачу в очередь и продолжает работу, а сервисы test-runner забирают задачи по мере своей готовности.
Это также позволяет масштабировать количество test-runner под нагрузку.

test-runner потребляет сообщения из kafka и вычитывает соответствующее решение и задачу из minio.
используя плагин конкретного языка, test-runner запускает docker контейнер для проверки решения.
в результате получаем вердикт по решению, которые отправляются в kafka.
вердикт по решению вычитывает web-server и кладёт результаты в БД.
так же вердикт возвращается к ожидающему пользователю.

Безопасность запуска недоверенного кода в docker контейнерах рассмотрена в моей статье.

### Сервис запуска кода task-runner

task-runner отдельный сервис, который позволяет запустить пользовательский код в docker контейнере вне решения задача 
с получением стандартного вывода. 
это удобно, когда и хочешь проверить какую-то функцию языка. 

### Сервис языковой поддержки lsp-manager

Для обеспечения превосходного опыта разработки решений используем LSP сервера.
Описать че это такое кратко.
Сервис lsp-manager будет управлять docker контейнерами с LSP серверами.
lsp-manager проксирует stdio контейнера <-> websocket с клиентом
На бэкенде будем запускать LSP сервер

## Масштабирование и репликация сервисов

Для обеспечения высокой доступности критически важно реплицировать сервисы.
Репликация webserver, test-runner, lsp-manager.

web-server будет реплицирован, поэтому вычитать сообщение с результатом может не тот сервис,
к которому сейчас подключен пользователь,
поэтому сообщение о результате также отправляется в Redis.

## AI Агент генерации задач

Современный AI агент должен содержать в себе и зачастую содержит в себе следующий набор функционала:

- Окно чата с удалением и редактированием сообщений, повторной генерацией ответа
- Сессии (отдельные чаты со своей историей сообщений)
- Чекпоинты, предпросмотр изменений
- Работа с файлами, консольные команды bash, grep, glob
- lsp, todo, webfetch, questions
- Правила в agents.md
- Agentic Workflow
- Advanced работа с контекстом
- Parallel tool calling
- Коннекторы к разнообразным провайдерам и разнообразным моделям
- MCP
- Skills
- RAG
- Субагенты
- И многое другое, что появится скором в будущем.

Работа по генерации задачи в большей степени заключается в генерации кода.
Сейчас появились сильные кодинг-агенты, которые уже содержат в себе перечисленный функционал. 
Тем более, разработчики этих кодинг-агенты будут дорабатывать их и добавлять новый функционал. 
Хорошим подходом в сфере программирования будет переиспользовать этот функционал.
Решение представлено на схеме. 

frontend предоставляет функции MCP серверу по протоколу WebSocket.

- platform-info
- exportThisProblem
- importThisProblem
- validate
- listProblems
- exportProblem

Локальный MCP сервер предоставляет эти же функции кодинг-агенту по протоколу MCP.

На фронтенде на экране редактирования задачи нажимает кнопку "экспорт" и выбирает свободную папку на своем компьютере.
В этой же папке запускает предпочитаемого кодинг-агента. Допустим, это opencode.
Экспорт происходит вместе с "навыком" для агента.
В навыке в файле SKILL.md описана вся необходимая информация:

- структура задачи
- схемы данных файлов задачи
- инструкция по инициализации проекта
- инструкция по созданию и доработке задачи

Пользователь может самостоятельно дорабатывать навык, если не получает от агента ожидаемого результата.

Пользователь начинает обсуждать с кодинг-агентом задачу по программированию.
Кодинг-агент определяет домен и читает скилл связанный с созданием задачи.
Агент работает с локальными файлами, запускает экспорт файлов на фронтенд, запускает валидацию.
Пользователь сохраняет задачу, если результат его устраивает.
Если агенту не хватает примеров, то он может получить список уже созданных пользователем задач, скачать эти задачи 
и использовать для улучшения качества генерации. 
Пользователь может изменить навык, чтобы агент всегда скачивал какие-то определенные референсы задач.

Данная архитектура агента позволяет не переизобретать готовые инструменты, не потерять в качестве генерации, 
и тем более не отстать в функционале через полгода без доработок.
