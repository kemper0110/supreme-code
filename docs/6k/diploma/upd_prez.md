# Обновленная структура презентации

Главная логика защиты: в бакалавриате была создана базовая онлайн-платформа, а в магистратуре она была переработана в более промышленную систему: с нормальной архитектурой, хранением данных, безопасностью, расширяемой поддержкой языков, AI-сценарием и наблюдаемостью.

Важная подача: не рассказывать презентацию как каталог фич. Каждый технический блок должен отвечать на вопрос комиссии: какую задачу я решал, какое проектное решение выбрал, что получилось в системе.

Целевой тайминг: 8:30-9:20. Слайд с выводами должен быть открыт до 9:30.

Терминология: `platform-core` - это презентационное название текущего модуля `web`, который раньше назывался `web-server`.

## Пользовательские сценарии

Время: 35-45 секунд.

Что на слайде:

Две горизонтальные дорожки или две колонки.

Сценарий решения задачи:

```text
каталог задач
  -> описание задачи
  -> редактор кода
  -> отправка решения
  -> результат проверки
```

Сценарий создания задачи:

```text
форма задачи
  -> языки
  -> тесты, решение, шаблон
  -> валидация
  -> публикация
```

Что подчеркнуть визуально:

- оба сценария сходятся в сущности "задача";
- решение задачи создает "решение" и "результат проверки";
- создание задачи требует файлов для каждого языка: тесты, эталонное решение, шаблон.

Что говорить:

> В платформе есть два основных пользовательских сценария. Первый - решение задачи: пользователь выбирает задачу из каталога, изучает описание, пишет код в редакторе, отправляет решение и получает результат проверки. Второй - создание задачи: автор заполняет описание, выбирает поддерживаемые языки, добавляет тесты, эталонное решение и шаблон, после чего задача валидируется и может быть опубликована. Эти сценарии определяют, какие сущности нужно хранить в базе данных и какие файловые артефакты нужно вынести в объектное хранилище.

Главная мысль:

> Сценарии решения и создания задачи задают требования к хранению данных.

Переход:

> Поэтому следующим шагом покажу, как эти сценарии отражаются в схеме БД и файловом хранилище.

## Хранение данных: PostgreSQL + MinIO

Время: 45-50 секунд.

Композиция слайда:

- слева маленькая схема БД на 7 таблиц;
- справа два дерева MinIO;
- стрелками связать сущности БД с файловыми артефактами.

Что показать в дереве MinIO:

```text
problems
/problems/{problemId}/languages/{languageId}/
  test.txt
  solution.txt
  solution-template.txt

solutions
/users/{userId}/problems/{problemId}/languages/{languageId}/solutions/{solutionId}/
  solution.txt
  logs.txt
  report.txt
```

Пояснение к файлам:

- `test.txt` - тесты задачи для конкретного языка;
- `solution.txt` в `problems` - эталонное решение;
- `solution-template.txt` - шаблон, который получает пользователь;
- `solution.txt` в `solutions` - отправленный пользователем код;
- `logs.txt` - вывод запуска проверки;
- `report.txt` - отчет тестового фреймворка после проверки.

Что говорить:

> В платформе данные разделены на две группы. Метаданные и связи между сущностями хранятся в PostgreSQL: пользователь, задача, язык, решение, результат проверки и связанные сущности. 
> Текстовые артефакты вынесены в MinIO. Для задачи хранятся тесты, эталонное решение и шаблон решения.
> При отправке решения оно сохраняется в виде файла. После проверки решения добавляются логи и отчет тестового фреймворка.

Главная мысль:

> База данных отвечает за предметную модель, MinIO - за файлы задач и проверок.

Переход:

> Теперь можно перейти к полной схеме компонентов, которые реализуют эти сценарии.

## Общая архитектура системы

Время: 55-60 секунд.

Что на слайде:

- большая схема со всеми компонентами;
- визуально выделить контуры:
  - `frontend`;
  - `platform-core`;
  - сервисы исполнения: `test-runner`, `task-runner`, `lsp-manager`;
  - хранилища и шины: PostgreSQL, MinIO, Kafka, Redis;
  - безопасность: Keycloak;
  - наблюдаемость: Prometheus, Grafana, Tempo, Loki.

Что говорить:

> На слайде показан полный состав системы. Я не буду разбирать каждый контейнер отдельно, а выделю основные контуры. Пользователь работает с frontend. Основная бизнес-логика сосредоточена в `platform-core`. Проверка решений, запуск произвольного кода и языковая поддержка вынесены в отдельные сервисы исполнения. Метаданные хранятся в PostgreSQL, файловые артефакты - в MinIO, асинхронная проверка построена через Kafka, а Redis используется для доставки событий пользователям при репликации `platform-core`.

Как встроить Keycloak:

> Отдельно отмечу безопасность. В первой версии авторизация была самописной. Для такой части системы это рискованно, поэтому в магистерской версии она заменена на Keycloak. Пользователь получает JWT, а `platform-core` проверяет доступ к операциям платформы по ролям и привилегиям. Детали кода здесь не показываю, потому что для архитектуры важно само разделение: управление пользователями и токенами вынесено в специализированный сервис.

Переход:

> На этой карте компонентов можно проследить основной технический процесс платформы - проверку решения.

## Проверка решений

Время: 65-75 секунд.

Что на слайде:

Отфильтрованная схема, а не вся большая архитектура:

```text
frontend
  -> platform-core xN
  -> PostgreSQL + MinIO
  -> Kafka
  -> test-runner xN
  -> Docker
  -> junit.xml / report.xml
  -> LanguageTester plugin
  -> Kafka
  -> platform-core xN
  -> Redis + SSE
  -> frontend
```

Что обязательно показать:

- `platform-core xN`, чтобы объяснить репликацию основного сервиса;
- `test-runner xN`, чтобы объяснить масштабирование проверки;
- Kafka между `platform-core` и `test-runner`;
- Docker как место запуска недоверенного кода;
- MinIO как источник тестов и решений;
- Redis/SSE как доставка результата пользователю.

Что говорить:

> Когда пользователь отправляет решение, `platform-core` создает запись о решении в PostgreSQL и сохраняет код в MinIO. После этого в Kafka отправляется сообщение на проверку. `test-runner` получает сообщение, загружает из MinIO тесты и пользовательское решение, затем запускает проверку в Docker-контейнере. Это важно, потому что пользовательский код является недоверенным. После завершения проверки `test-runner` получает отчет тестового фреймворка, преобразует его в единый формат результата и отправляет результат обратно через Kafka. `platform-core` сохраняет результат и публикует уведомление через Redis, после чего frontend получает событие по SSE.

Как объяснить Redis:

> Redis нужен из-за репликации `platform-core`. Результат из Kafka может получить не тот экземпляр, к которому сейчас подключен пользователь. Поэтому событие публикуется в общий Redis-канал, и нужный экземпляр доставляет его пользователю.

Главная мысль:

> Проверка решений стала асинхронной, масштабируемой и изолированной от пользовательского кода.

Переход:

> Следующий вопрос - как привести результаты разных языков и тестовых фреймворков к единому виду.

## Единый формат результата проверки

Время: 45-55 секунд.

Что на слайде:

Схема преобразования:

```text
pytest / JUnit / Jest / CTest
  -> junit.xml или report.xml
  -> LanguageTester plugin
  -> TestExecutionResult
  -> результат в интерфейсе
```

Маленький фрагмент `junit.xml` или `report.xml`:

```xml
<testsuite tests="3" failures="1" errors="0">
  <testcase classname="SolutionTest" name="case_1"/>
  <testcase classname="SolutionTest" name="case_2">
    <failure message="expected 42, got 41"/>
  </testcase>
</testsuite>
```

Минимальный фрагмент интерфейса плагина:

```java
TestExecutionResult verdict(String report, int exitCode);
```

Что говорить:

> У разных языков используются разные инструменты проверки: например, `pytest`, `JUnit`, `Jest` или `CTest`. Но для платформы результат должен выглядеть одинаково: сколько тестов выполнено, сколько упало, какие ошибки произошли, засчитано ли решение. Поэтому после запуска тестов отчет в формате `junit.xml` или похожем XML преобразуется через языковой плагин в единый `TestExecutionResult`.

Главная мысль:

> Поддержка нового тестового фреймворка не требует менять `platform-core`: достаточно добавить или выбрать плагин преобразования результата.

Переход:

> Но одного преобразования результата мало. Для языка нужно еще описать, как его запускать и какие файлы куда класть. Это вынесено в конфигурацию платформы.

## Конфигурация языков

Время: 55-60 секунд.

Что на слайде:

Фрагмент `platform.yaml`, лучше на примере Python или TypeScript. Не показывать весь файл.

```yaml
python:
  name: Python
  monacoLanguageId: python
  testerConfig:
    imageName: supremecode-python
    cmd: ["pytest", "--junitxml=report.xml"]
    testPath: /usr/test/test_solution.py
    solutionPath: /usr/test/solution.py
    reportPath: /usr/test/report.xml
    verdictClassName: org.supremecode.commonplugins.PyTestLanguageTester
  runnerConfig:
    image: supremecode-python
    cmd: ["python", "main.py"]
    filePath: main.py
  lspConfig:
    image: supremecode-python-lsp
    cmd: ["pyright-langserver", "--stdio"]
```

Что подсветить цветом или рамками:

- `testerConfig` - проверка решений;
- `runnerConfig` - запуск кода в песочнице;
- `lspConfig` - языковой сервер для редактора;
- `verdictClassName` - выбор плагина для преобразования отчета.

Что говорить:

> Конфигурация языка собрана в одном месте. Для проверки указывается Docker-образ, команда запуска тестов, пути к файлам решения и тестов, путь к отчету и класс плагина, который преобразует отчет в единый результат. Отдельно задается конфигурация запуска кода в песочнице и конфигурация LSP. Поэтому новый язык подключается не переписыванием ядра платформы, а добавлением конфигурации, Docker-образов и при необходимости нового плагина.

Главная мысль:

> Языки стали конфигурируемой частью платформы, а не набором условий в коде `platform-core`.

Переход:

> Эта же конфигурация используется не только для проверки, но и для улучшения редактора кода через LSP.

## LSP для редактора кода

Время: 40-50 секунд.

Что на слайде:

Схема без gif:

```text
Monaco editor
  -> WebSocket
  -> lsp-manager
  -> Docker container
  -> language server stdio
```

Подписи на схеме:

- frontend работает с Monaco;
- соединение идет по WebSocket;
- `lsp-manager` создает контейнер с LSP-сервером;
- stdio LSP-сервера проксируется в WebSocket;
- контейнер содержит окружение языка и зависимости.

Что говорить:

> В базовой версии редактор фактически давал только ввод кода и подсветку. В магистерской версии добавлена интеграция с LSP. Платформа не реализует анализ кода самостоятельно, а использует существующие language servers. Для пользователя это дает автодополнение, диагностику и более привычный опыт разработки. На серверной стороне `lsp-manager` запускает LSP-сервер в Docker-контейнере и проксирует его стандартный ввод-вывод в WebSocket-соединение с редактором.

Переход:

> Следующая доработка связана уже не с решением задач, а с их созданием и редактированием.

## AI-редактирование задач

Время: 55-65 секунд.

Что на слайде:

Схема:

```text
frontend
  <-> локальный MCP server
  <-> coding agent
  <-> директория задачи на диске
       - файлы задачи
       - skill / инструкции
```

Что обязательно подписать:

- задача экспортируется из frontend в директорию;
- вместе с задачей экспортируется skill с описанием формата и правил работы;
- coding agent работает с файлами локально;
- через MCP вызывается валидация и импорт обратно во frontend.

Что говорить:

> Создание задач по программированию часто требует написать описание, тесты, эталонное решение и шаблоны для разных языков. Вместо того чтобы писать собственного AI-агента внутри платформы, я использовал другой подход. Платформа экспортирует задачу как набор файлов на диск и добавляет skill с описанием структуры задачи. Пользователь запускает привычного coding agent в этой директории. Агент редактирует файлы, а через MCP может вызывать функции платформы, например валидацию задачи и импорт результата обратно во frontend.

Главная мысль:

> Платформа не конкурирует с существующими AI-инструментами, а дает им понятный интерфейс к задачам и валидации.

Переход:

> Когда система состоит из нескольких сервисов и асинхронных процессов, важно видеть, что с ней происходит при работе.

## Наблюдаемость

Время: 45-55 секунд.

Что на слайде:

- крупно: бизнесовые метрики или несколько ключевых графиков;
- рядом: выделенный trace с прямоугольниками по сервисам;
- маленькая вставка: логи, отфильтрованные по `trace_id`.

Лучше использовать:

- `docs/6k/diploma/prez/observability/business-metrics.png`;
- `docs/6k/diploma/prez/observability/trace-highlighted.png`;
- `docs/6k/diploma/prez/observability/loki-logs.png` как маленькую вставку, если помещается.

Что говорить:

> Для сопровождения платформы внедрены инструменты наблюдаемости. Это важно для эксплуатации системы. Метрики показывают общее состояние и нагрузку, например количество отправок решений и результаты проверок. Трассировка позволяет увидеть путь конкретной операции через `platform-core`, Kafka, `test-runner`, MinIO и Docker. Логи можно отфильтровать по `trace_id`, чтобы восстановить контекст конкретного запроса.

Короткая формула:

> Метрики показывают, что проблема есть. Trace показывает, где она возникла. Логи помогают понять, почему.

Не делать:

- не пытаться заставить комиссию читать весь trace;
- не углубляться в OpenTelemetry, Loki, Tempo и Prometheus как отдельные технологии;
- не оправдываться, что "архитектурно ничего сложного". Формулировка: "подключена эксплуатационная наблюдаемость".

Переход:

> После проектных и технических решений покажу итоговый пользовательский сценарий работы платформы.

## Демонстрация работы

Время: 60-75 секунд.

Формат: не живое демо, а storyboard из скриншотов и gif.

Что на слайде:

1. создание или редактирование задачи;
2. экспорт задачи и работа AI-агента;
3. gif работы LSP в редакторе;
4. отправка решения на проверку;
5. получение результата проверки;
6. trace или метрики как подтверждение обработки.

Что говорить:

> В демонстрационном сценарии создадим "Лабораторную по ИИ №1" на python с внешней библиотекой numpy. 
> Пишем описание. Coding agent помогает доработать тесты, решение или шаблон. 
> Затем пользователь решает задачу в редакторе. В написании кода ему помогает языковой сервер. 
> Пользователь отправляет решение на проверку и получает результат асинхронно. 
> За этим пользовательским сценарием стоит уже показанная архитектура: хранение файлов в MinIO, проверка через Kafka и `test-runner`, запуск в Docker и сохранение результата.

Как построить визуально:

- не делать 6 мелких скриншотов одинакового размера;
- главный кадр - результат или редактор;
- остальные кадры сделать меньшими шагами процесса;
- gif LSP лучше поставить рядом с шагом решения задачи.

Переход:

> В завершение сопоставлю полученный результат с поставленными задачами.

## Выводы

Время: 35-45 секунд.

Что на слайде:

Задача -> результат.

- переработана архитектура платформы: выделены `platform-core`, сервисы исполнения и инфраструктурные компоненты;
- реализовано разделение хранения: метаданные в PostgreSQL, файлы задач и решений в MinIO;
- проверка решений организована асинхронно через Kafka и изолирована в Docker;
- добавлена расширяемая языковая подсистема через `platform.yaml`, Docker-образы и плагины преобразования результатов;
- добавлена LSP-поддержка редактора кода;
- реализован сценарий AI-редактирования задач через MCP, локальную директорию и skill;
- подключены Keycloak и инструменты наблюдаемости;
- работа платформы продемонстрирована на пользовательском сценарии.

Что говорить:

> В результате была не просто добавлена отдельная функциональность, а переработана платформа как система. Архитектура стала более модульной, хранение разделено между БД и объектным хранилищем, проверка решений стала асинхронной и контейнеризированной, языки вынесены в конфигурацию и плагины, редактор получил LSP-поддержку, а создание задач может выполняться с помощью AI-агента. Также добавлены готовая система авторизации и наблюдаемость, что важно для сопровождения платформы.

Финальная фраза:

> Таким образом, поставленные задачи выполнены, а платформа подготовлена к дальнейшему расширению и использованию в учебных сценариях.

# Проверка тайминга

Ориентировочный суммарный тайминг:

- титульный слайд: 0:15;
- актуальность: 0:35;
- цель и задачи: 0:55;
- пользовательские сценарии: 0:40;
- хранение данных: 0:45;
- общая архитектура системы: 0:55;
- проверка решений: 1:05;
- единый формат результата проверки: 0:45;
- конфигурация языков: 0:55;
- LSP для редактора кода: 0:45;
- AI-редактирование задач: 0:55;
- наблюдаемость: 0:45;
- демонстрация работы: 1:05;
- выводы: 0:40.

Итого: примерно 10 минут. При прогоне, если не влезает, сокращать в таком порядке:

1. наблюдаемость - оставить только формулу "метрики -> trace -> логи";
2. AI-редактирование задач - сократить подробности про skill;
3. конфигурация языков - не читать YAML, а назвать только подсвеченные поля;
4. хранение данных - не объяснять каждый файл в MinIO;
5. пользовательские сценарии - оставить только названия двух сценариев и их связь с хранением данных.

Желательный прогон: 8:30-9:20. На 9:30 уже должен быть открыт слайд с выводами.

# Общие правила подачи

- Не говорить "я просто прикрутил", "кое-как подключил", "там ничего особенного". Заменять на "интегрировано", "подключена эксплуатационная наблюдаемость", "выбрано готовое решение".
- В технических слайдах про проверку, единый формат результата и конфигурацию языков текст должен быть крупным: показывать только фрагменты схемы, XML и YAML.
- Большую схему использовать один раз как карту системы. Дальше показывать отфильтрованные маршруты.
