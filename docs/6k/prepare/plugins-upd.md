
# Конфигурация и плагинизация

Развитие платформы для решения программных задач связано с необходимостью поддерживать несколько языков программирования, разные способы запуска пользовательского кода и различные тестовые фреймворки. При этом ядро системы не должно изменяться каждый раз, когда добавляется новый язык или обновляется способ проверки решений. Для решения этой задачи в проекте используется конфигурация платформы и механизм языковых плагинов. Конфигурация платформы описывает внешние свойства языка и параметры запуска контейнеров, а языковой плагин отвечает за преобразование результата тестирования во внутренний формат результата.

Главным источником сведений о поддерживаемых языках является файл «platform.yaml». Его структура в программном коде описана набором классов конфигурации, расположенных в модуле shared. Корневым объектом является PlatformConfig, который содержит поле languages. Это поле представляет собой отображение, где ключом является идентификатор языка, а значением является объект LanguageConfig. Например, ключ python обозначает конфигурацию языка Python и используется сервисами как устойчивое имя языка при обмене данными.

Структура конфигурации платформы показана на UML-диаграмме классов. Диаграмма отражает, что LanguageConfig объединяет общие сведения о языке и три необязательных раздела: TesterConfig, RunnerConfig и LspConfig. Такой состав позволяет описывать язык единым образом, но использовать разные части описания в разных сервисах. Фронтенд обращается к сведениям для отображения языка и настройки редактора, сервис test-runner использует параметры проверки, сервис task-runner использует параметры запуска пользовательской программы, а сервис языковой поддержки использует параметры запуска LSP-сервера. Поэтому «platform.yaml» выступает не как набор несвязанных настроек, а как единая модель платформенной конфигурации.

Дополнительно такая модель упрощает сопровождение проекта, поскольку изменение параметров языка выполняется в одном месте и затем одинаково интерпретируется всеми компонентами, которые читают общую конфигурацию платформы.

Поля объекта LanguageConfig описывают пользовательское представление языка, работу редактора и связи с сервисными разделами:

- поле name задает отображаемое название языка;
- поле iconPath указывает путь к значку, который может быть показан в интерфейсе выбора языка;
- поле monacoLanguageId связывает язык платформы с идентификатором языка в редакторе Monaco, благодаря чему редактор применяет подсветку синтаксиса и базовые правила работы с текстом;
- поле monacoFile задает путь к файлу в модели редактора и позволяет согласовать виртуальный путь с рабочим окружением языка;
- поле extensions задает список расширений файлов, которые относятся к данному языку;
- поле ephemeralFileName определяет имя временного файла, создаваемого для решения или демонстрационного запуска;
- поле playgroundInitialCode содержит начальный текст программы, который пользователь видит в режиме песочницы;
- поле testerConfig содержит параметры проверки решений, если для языка поддерживается тестирование;
- поле runnerConfig содержит параметры запуска пользовательского кода, если для языка поддерживается режим выполнения;
- поле lspConfig содержит параметры сервера языковой поддержки, если для языка доступны редакторные подсказки и диагностика;

Раздел testerConfig предназначен для сервиса test-runner. Этот сервис получает решение и тесты, помещает их в контейнер и запускает тестовый фреймворк. В данном разделе используются следующие поля:

поле imageName задает «Docker-образ» для проверки решения;
поле cmd определяет команду запуска тестов, и в примере Python эта команда запускает pytest с формированием отчета «report.xml»;
поле testPath задает путь, по которому внутри контейнера размещается файл с тестами;
поле solutionPath задает путь, по которому внутри контейнера размещается файл с пользовательским решением;
поле reportPath указывает, откуда после завершения проверки нужно забрать отчет;
поле verdictClassName задает полное имя класса языкового плагина, который будет интерпретировать полученный отчет;

Раздел runnerConfig используется сервисом task-runner, который отвечает не за проверку решения тестами, а за простой запуск пользовательского кода. В данном разделе используются следующие поля:

поле image указывает «Docker-образ», в котором должна выполняться программа;
поле cmd задает команду запуска пользовательского кода;
поле filePath определяет путь к основному файлу внутри рабочей директории контейнера;

Для Python таким файлом является «main.py», а команда запуска вызывает интерпретатор Python для этого файла. Благодаря этому сервис task-runner не содержит специальных условий для каждого языка, а использует декларативные параметры из общей конфигурации.

Раздел lspConfig описывает работу сервера языковой поддержки. Этот сервер нужен для функций редактора, связанных с анализом кода, диагностикой и подсказками. В данном разделе используются следующие поля:

- поле image указывает «Docker-образ» с установленным LSP-сервером;
- поле cmd задает команду запуска сервера языковой поддержки;

В примере Python используется «pyright-langserver» в режиме обмена данными через стандартные потоки. Таким образом, добавление редакторной поддержки также сводится к изменению конфигурации, а не к доработке клиентского приложения или инфраструктурного сервиса.

Важной частью конфигурации является поле verdictClassName. Это поле связывает декларативное описание языка с плагинной частью системы. При выполнении проверки сервис test-runner запускает тестовый фреймворк в контейнере, получает отчет и код завершения процесса, после чего должен привести результат к единой модели. Такая унификация необходима потому, что разные тестовые фреймворки по-разному описывают ошибки, пропущенные тесты, имена наборов тестов и длительность выполнения. Даже когда несколько инструментов формируют отчет в формате «junit.xml», структура элементов, атрибуты и смысл отдельных полей могут немного отличаться. Кроме того, архитектура не должна зависеть только от «junit.xml», так как новый язык или новый фреймворк может возвращать результат в другом формате.

Именно для этой задачи используется языковой плагин. Плагин получает текст отчета и код завершения процесса, анализирует их с учетом особенностей конкретного тестового фреймворка и возвращает внутренний формат результата. Для Python таким плагином является класс org.supremecode.commonplugins.PyTestLanguageTester. Его имя записывается в verdictClassName, поэтому ядро test-runner не выбирает обработчик по набору условных операторов и не содержит знания о формате отчета PyTest. Ядро знает только общий контракт, а конкретная логика разбора находится в подключаемой реализации.

Контракт для языковых плагинов расположен в модуле plugin-sdk. Этот модуль можно рассматривать как границу между стабильной частью платформы и расширениями для отдельных языков. В нем определен интерфейс LanguageTester, который содержит функцию verdict. Эта функция принимает отчет тестового фреймворка и код завершения процесса, а возвращает объект TestExecutionResult. Следовательно, любой языковой плагин обязан реализовать единый способ получения вердикта, но внутри реализации может использовать собственный алгоритм разбора отчета.

Модель TestExecutionResult задает общий результат выполнения тестов и включает следующие поля:

поле summary содержит объект Summary с агрегированными счетчиками проверки;
поле testCases содержит список объектов TestCase, каждый из которых описывает отдельный тестовый случай;
поле solved содержит логический признак успешного решения задачи;

Объект Summary хранит агрегированные счетчики:

поле total задает общее число тестов;
поле passed задает число успешных тестов;
поле failures задает число неуспешных проверок;
поле errors задает число ошибок;
поле skipped задает число пропущенных тестов;

Объект TestCase описывает отдельный тестовый случай:

поле name задает имя теста;
поле suiteName задает имя набора тестов;
поле status задает статус тестового случая;
поле message содержит диагностическое сообщение;
поле durationMs задает длительность выполнения в миллисекундах;

Перечисление TestStatus ограничивает набор возможных статусов значениями PASSED, FAILED, ERROR и SKIPPED. Благодаря такой модели остальные компоненты платформы получают результат проверки в одинаковом виде независимо от языка программирования и используемого фреймворка.

Подключение языковых плагинов выполняется через механизм «ServiceLoader/SPI». Реализация интерфейса LanguageTester помещается в classpath сервиса test-runner. Затем полное имя класса указывается в файле «META-INF/services/org.supremecode.pluginsdk.LanguageTester». При запуске сервис загружает все объявленные реализации через стандартный механизм Java SPI и формирует сопоставление между полным именем класса и объектом плагина. Когда для языка в конфигурации указано значение verdictClassName, сервис находит соответствующий экземпляр плагина и использует его для построения результата проверки.

Такое устройство делает расширение платформы более упорядоченным. Для добавления языка нужно описать его в «platform.yaml», подготовить «Docker-образ» для запуска решений, при необходимости подготовить «Docker-образ» с LSP-сервером и реализовать языковой плагин для обработки отчета тестового фреймворка. После этого новый обработчик регистрируется через «META-INF/services/org.supremecode.pluginsdk.LanguageTester», а его полное имя записывается в verdictClassName. В результате ядро test-runner, сервис task-runner и клиентская часть продолжают работать по прежним правилам, а особенности языка остаются изолированными в конфигурации и плагине.

Единая конфигурация платформы также снижает связанность между сервисами. Фронтенд использует сведения, необходимые для отображения языка и настройки редактора. Сервис task-runner использует параметры запуска пользовательской программы. Сервис test-runner использует параметры проверки и имя языкового плагина. Сервис языковой поддержки использует параметры запуска соответствующего серверного процесса. При этом все эти компоненты опираются на одно описание языка и не дублируют сведения о нем в собственном коде. Поэтому конфигурация и плагинизация вместе образуют механизм расширяемости, который соответствует микросервисной архитектуре платформы и позволяет поддерживать новые языки без изменения ядра существующих сервисов.
